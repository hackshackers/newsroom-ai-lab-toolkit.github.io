# ♻️ Iterative Development

Earlier stages of the Design Thinking process focus on observing, listening, defining the problem clearly, and clarifying what success should look like before jumping to the solution itself.

With a clearly defined problem in hand, you are ready to start building toward a solution. The **Iterate** stage is where you move from definition to action - generating ideas, prototyping quickly, and testing with real users.

Each iteration cycle brings you closer to a solution that actually works.

## Now it's time to start building — but *what*?

…we don't know exactly, yet.… and that's intentional!

When you try to define the entire solution upfront, you have to make a lot of assumptions about the most effective approaches \- **early guesses can harden into plans.**

If we pretend we have solution certainty too early, we'll most likely end up **very carefully building something ✨beautiful✨ that does not actually serve the people it was meant to help.**

## ♻️ Iterative Development says:

1. We know enough to take a thoughtful next step.
2. We do not know enough to assume the whole plan is right.
3. We will test, learn, adjust, and keep moving towards the solution.

A useful iteration is **a focused cycle that helps the team learn something important**.

Rapid, grounded cycles add up to a solution that is much more likely to actually solve the core problem.

## ♻️ Iterative Development contrasts with ⛲ Waterfall Development

### ⛲ Waterfall Development involves non-iterative, incremental development which:

- Begins with a **fully comprehensive solution specification**
- Relies on a pre-defined, sequential development process that flows like a waterfall through all phases of a project with **each phase completely wrapping up before the next phase begins**
- Involves **stakeholders only at the start and the end** of the project

### ⛲ Waterfall is essential when the blueprint is knowable

When the solution is already well understood, there's little to learn along the way.
